# Renderer Architecture
generated: 1749513600

---

The renderer is split across five focused modules under `src-tauri/src/renderer/`. Each module owns one concern. They share state through `Arc<Mutex<>>` handles that are set up at startup and passed into the render thread.

---

## Modules

### `renderer/mod.rs`; Orchestration
The entry point. Defines the shared state types and owns the render loop.

**Key types:**

- `NyxRenderer` - the public handle held by Tauri. Wraps `hwnd`, `state`, `camera_input`, and `undo` as `Arc<Mutex<>>` fields.
- `SceneState` - the live scene state. Holds the command list, active profile, physics world, viewport bounds, camera, selection, gizmo mode, and a `dirty` flag.
- `CameraInput` - accumulator for orbit, pan, zoom, and WASD deltas between frames.
- `UndoHistory` - two `Vec<Vec<serde_json::Value>>` stacks (undo and redo).

**Render loop (`RenderLoop`):**

Runs on its own thread. Each tick it:
1. Reads camera input and applies it to the orbital camera.
2. Checks the `dirty` flag, if set, calls `LoadCommands()` and `LoadGizmo()` to update GPU buffers.
3. Calls `render()` on `SceneRenderer` — outside the state lock, so a vsync wait in `present()` never blocks input/command handlers. Frames are never skipped when new commands land mid-frame; the next iteration picks them up (the previous skip-on-dirty behavior caused visible stutter during drags).
4. Paces frames against `MAX_FPS` (240): sleeps whatever remains of the frame budget, measured from loop start. Idle ticks sleep longer (14 ms visible, 16 ms hidden).

Scene *rebuilds* (command snapshot + GPU upload) have a separate `MAX_REBUILD_FPS` (120) cadence: physics keeps the scene dirty every iteration while bodies are awake, and uncapped rebuilds monopolize the state lock that live-scene and input commands need — which starved script playback (tweens). Camera-only redraws still pace at 240; a deferred rebuild lands within ~8 ms.

**Live scenes (script playback):** a backend thread (`LiveSceneLoop` in `scene_runner.rs`, started/stopped via the `start_live_scene` / `stop_live_scene` commands) re-runs the scene script at 60 fps with the wall-clock elapsed time and applies the resulting commands directly to renderer state — no IPC round-trips per tick. It pauses itself while the viewport is hidden (`SceneState.visible`) or within 140 ms of an object edit.

**Edit gate (`SceneState.last_edit_interaction`):** only object edits arm it — gizmo drags (native WndProc and the `renderer_*_drag` / `renderer_set_part_properties` commands). Camera navigation (orbit/pan/zoom/WASD fly) deliberately does not: arming it on camera moves froze script playback whenever the camera was in motion.

Colors in commands are `{R, G, B}` objects (positions/sizes are `{X, Y, Z}`); the renderer reads them via `ReadColor()`, covered by a regression test after the keys got mixed up once.

`ProcessMetaCommands()` runs on dirty to handle `SetSkybox` and `SetCamera` commands before geometry is loaded. `UpdateCameraClip()` adjusts the far plane dynamically based on scene bounds.

### `renderer/scene.rs` -- GPU Rendering
Owns all wgpu resources and executes draw calls.

**Key types:**

- `SceneRenderer`; holds the four render pipelines, all GPU buffers, MSAA texture (4x), and depth texture (32-bit float).
- `ShapeBatch`; one per primitive shape — unit-space geometry built once at startup plus a growable per-instance buffer (starts at 256, doubles on demand).
- `CachedMesh`; an `AddMesh` upload (local-space geometry + a single-entry instance buffer) cached alongside the command JSON that produced it.

**Pipelines and their shaders:**

| Pipeline | Shader | What it draws |
|---|---|---|
| `sky_pipeline` | `sky.wgsl` | Fullscreen procedural sky gradient + sun glow |
| `part_pipeline` | `roblox.wgsl` | All lit geometry: primitive parts instanced per shape (Block, Sphere, Cylinder, Cone, Wedge, Torus) and `AddMesh` draws as single-instance batches |
| `gizmo_pipeline` | `gizmo.wgsl` | Gizmo lines and selection outlines |
| `grid_pipeline` | `grid.wgsl` | Floor grid quad |

All pipelines share `common.wgsl`, a prelude concatenated at pipeline creation that holds the camera uniform, light rig constants, ACES tonemapping, and `lit_surface()`. Pipeline descriptors are built by a single `MakePipeline()` helper.

**Color management:** the surface format is sRGB. Shaders work in linear space and let the hardware apply the transfer curve; nothing gamma-encodes manually. The `SetSkybox` color is converted sRGB→linear once per frame on the CPU before it reaches the clear color and camera uniform.

**Scene load (`LoadCommands`):**

Iterates the command list. For each command:
- `AddPart`, appends an `Instance` (position/size/color/rotation) to the batch for its shape. Unknown shape names fall back to the Block batch. Instance buffers grow; nothing is dropped.
- `AddMesh`, looks up the mesh cache by command `Id`, then takes the cheapest tier that applies:
  1. command JSON identical — nothing happens;
  2. geometry fields (`Vertices`/`Indices`/`Normals`) identical but transform or color changed (a drag) — one 64-byte instance-buffer write;
  3. geometry changed — vertex/index buffers rebuild.

  Stale entries are evicted after each load.

Because physics and drags mark the scene dirty every step, these tiers are what keep interaction smooth: moving any object — part or detailed mesh — is a pure instance write, and geometry only ever uploads when it actually changes.

**Render pass order:**

1. Sky gradient (depth off, fullscreen triangle)
2. Instanced shape batches, then cached mesh draws (part pipeline)
3. Floor grid (grid pipeline)
4. Gizmo lines and selection outline (gizmo pipeline)

All draws happen in one MSAA pass that resolves to the surface, with depth testing enabled for geometry.

### `renderer/mesh.rs` -- Geometry
Produces `Vertex` arrays and creates `wgpu::Buffer` objects.

**Key types:**

- `Vertex` - position and normal (all lit geometry, parts and meshes alike).
- `Instance` - position, size, color, and a quaternion rotation (consumed by the instanced shader; built from a command via `ReadInstance()`).
- `MeshDraw` - vertex/index buffers plus a single-entry instance buffer, ready to bind.

**Primitive shape generators:**

`CubeMesh`, `SphereMesh`, `CylinderMesh`, `ConeMesh`, `WedgeMesh`, `TorusMesh` all produce unit-space `(Vec<Vertex>, Vec<u32>)` with no color baked in. `ShapeIndex()` maps a shape name string to a batch slot (unknown names map to Block) and `UnitShapeMesh()` dispatches to the generator.

**Transform path:**

Nothing bakes transforms into vertices. `BuildMeshGeometry()` produces local-space `AddMesh` geometry (computing normals when the command omits them), `MakeMeshDraw()` uploads it once, and `ReadInstance()` packs the command's Position/Size/Color/CFrame into the `Instance` that the vertex shader applies — identically for parts and meshes.

### `renderer/gizmo.rs` - Transform Visualization
Produces `GizmoVertex` lines for the active gizmo and a `GridVertex` quad for the floor grid.

**Key types:**

- `GizmoVertex` - position and color (RGB per axis: X = red, Y = green, Z = blue).
- `GridVertex` - position only.

**Gizmo modes:**

`BuildGizmoVerts()` branches on the gizmo mode string:
- `"move"` - three axis lines with arrow tips, plus a bounding box outline.
- `"rotate"` - three orthogonal rings at the object center.
- `"scale"` - three axis lines with cube handles, plus a bounding box outline.

In all modes, a yellow bounding box is appended. If a face is selected, `AppendSelectedFaceOutline()` draws a yellow outline over the selected triangle.

`GizmoMetrics()` scales axis length, handle size, and ring radius relative to the object's bounding size so the gizmo always looks proportional.

### `renderer/window.rs` - Input and OS Window
Creates and manages the child `HWND` that the wgpu surface renders into.

**Startup:**

`CreateChildWindow()` creates a `WS_CHILD` window sized and positioned over the React webview's viewport element. `InitViewportInput()` stores the `Arc<Mutex<>>` state pointers in thread-local statics (`VP_CAM`, `VP_STATE`, `VP_UNDO`, `VP_APP`) so `WndProc` can reach them without passing handles through Windows callbacks.

**Input handling (`WndProc`):**

| Message | Action |
|---|---|
| `WM_LBUTTONDOWN` | `GizmoHit()` to test axis under cursor, then starts orbit or gizmo drag |
| `WM_MOUSEMOVE` | Accumulates camera orbit/pan delta or calls `DoMoveDrag` / `DoRotateDrag` / `DoScaleDrag` |
| `WM_LBUTTONUP` | `ClickSelect()` if not dragged, finalizes drag, emits `"vp-selected"` Tauri event |
| `WM_RBUTTONDOWN` | Starts pan mode, arms fly timer for WASD |
| `WM_MOUSEWHEEL` | Accumulates zoom |
| `WM_TIMER` | Polls WASD keys for continuous camera flight |

**Z-order:**

`SetZOrder()`, `DetachWindow()`, and `AttachWindow()` manage the child window stacking so the wgpu surface stays behind React panels (settings, notes, etc.) when they open. This is the "z-order requirement" referenced in the renderer memory.

**Selection:**

`ClickSelect()` does ray-AABB intersection against all parts in the command list.
`PickMeshFace()` does ray-triangle intersection against the active mesh draw's geometry.
Both emit Tauri events (`"vp-selected"`, `"vp-face-selected"`) so the React side can update the properties bar.

## End-to-End Scene Load

```
TypeScript: RendererService.LoadScene({ Commands, Profile })
  -> Tauri command: renderer_load_scene
  -> SceneState.commands = Commands, .profile = Profile, .dirty = true

Render thread detects dirty:
  -> ProcessMetaCommands()   (SetSkybox, SetCamera)
  -> SceneRenderer::LoadCommands()
       -> AddPart  -> Instance in the ShapeBatch for its shape
       -> AddMesh  -> unchanged? reuse
                      moved/recolored? instance write
                      new geometry? BuildMeshGeometry -> MeshDraw
  -> SceneRenderer::LoadGizmo()  (rebuild gizmo lines for selected object)
  -> dirty = false
  -> render()
```

## Shader Files

All shaders are embedded at compile time via `include_str!`.

| File | Role |
|---|---|
| `common.wgsl` | Shared prelude: camera uniform, light rig, ACES tonemap, `lit_surface()`. Prepended to every other shader at pipeline creation (WGSL has no include). |
| `sky.wgsl` | Procedural sky gradient (zenith/horizon/ground bands derived from the `SetSkybox` color) plus a sun glow aligned with the key light. |
| `roblox.wgsl` | All lit geometry — instanced parts and single-instance meshes. |
| `gizmo.wgsl` | Gizmo lines, unlit. |
| `grid.wgsl` | Floor grid with per-axis anti-aliasing and derivative-based minor-line fade. |

All live in `src-tauri/src/renderer/shaders/` and are embedded at compile time via `include_str!`. A headless test (`renderer::scene::tests::ShadersAndPipelinesValidate`) builds every pipeline against a real device so shader errors surface in `cargo test` instead of at app launch.
